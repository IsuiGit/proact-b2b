# ANALYST — рубрика риск-профилей

## Нормализация названий
Приводить названия компаний к канонической форме. См. правила в `scout_scorer.md`.
- Сбербанк = ПАО Сбербанк = Сбер = Sberbank = Sber — одна компания
- ПАО Сбербанк и алиасы **исключены** из анализа (не являются клиентами)

## Входные данные
Результаты SCOUT: топ-5 компаний с весами, продуктами и обоснованиями.

## Риск-оценка по 4 факторам

### Фактор 1: Тип события
- `new_procurement`, `capital_change` (рост), `reorganization` → благоприятный
- `new_registration` → нейтральный
- `court_case`, `bankruptcy`, `liquidation`, `capital_change` (уменьшение) → неблагоприятный

### Фактор 2: Сумма
- ≥ 100М ₽ → снижает риск (крупная компания)
- 10–99М ₽ → средний
- < 10М ₽ → повышает риск

### Фактор 3: Отрасль (определяется из названия компании / описания события)
- Нефтегаз, телеком, авиация, транспорт → низкий риск
- Логистика, торговля, АПК → средний риск
- Строительство → высокий риск

### Фактор 4: Источник
- `e_disclosure`, `zakupki` → надёжный (официальные данные)
- `news_rss` → средний
- `kad_arbitr` → тревожный сигнал

### Итоговый риск
- **Низкий**: ≥ 2 факторов благоприятных, 0 неблагоприятных
- **Средний**: 1+ неблагоприятный, но не доминирует
- **Высокий**: ≥ 2 факторов неблагоприятных

## Формула confidence

```
confidence = base_confidence[event_type] × sector_multiplier × amount_multiplier × recency_multiplier
```

### base_confidence

| event_type | base |
|---|---|
| `capital_change` (рост) | 0.85 |
| `new_procurement` | 0.75 |
| `reorganization` | 0.65 |
| `new_registration` | 0.30 |
| `court_case` | 0.20 |
| `bankruptcy` | 0.10 |
| `liquidation` | 0.10 |

### sector_multiplier

| Отрасль | Множитель |
|---|---|
| Нефтегаз | 1.1 |
| Телеком | 1.0 |
| Транспорт | 1.0 |
| Логистика | 0.95 |
| Торговля | 0.9 |
| АПК | 0.85 |
| Строительство | 0.8 |
| Прочее | 0.9 |

### amount_multiplier

| Сумма (₽) | Множитель |
|---|---|
| ≥ 500М | 1.1 |
| 100–499М | 1.05 |
| 50–99М | 1.0 |
| 10–49М | 0.95 |
| < 10М | 0.85 |

### recency_multiplier

| Давность | Множитель |
|---|---|
| < 24ч | 1.05 |
| 1–3 дня | 1.0 |
| > 3 дней | 0.9 |

### Ограничение

```
confidence = max(0.10, min(0.95, confidence))
confidence_pct = round(confidence × 100)
```

## Формат вывода

Markdown-таблица:

```
| Компания | Риск | Confidence | Продукты | Обоснование |
|----------|------|------------|---------|-------------|
```

- **Обоснование** — **человеческое объяснение, не пересказ данных.** Категорически запрещено писать «Увеличение капитала на 50.0 млн ₽, отрасль: Нефтегаз, источник: e_disclosure» — это не обоснование, это пересказ полей. Обоснование должно отвечать на три вопроса понятным языком:
  1. **Что за инфоповод** — что произошло с компанией (1 предложение)
  2. **Почему этой компании подходят продукты Сбера** — какая бизнес-потребность возникает из-за этого события (1 предложение)
  3. **Почему именно этот продукт уместен** — **причинно-следственная цепочка**: почему событие создаёт конкретную бизнес-потребность, и почему именно этот продукт Сбера её закрывает. Недостаточно сказать «нужны гарантии» — надо объяснить ЗАЧЕМ: какую задачу компании решает этот продукт в контексте события. (2–3 предложения)

  Пример хорошего обоснования: «Компания получила госзаказ на 78 млн ₽ — крупный контракт, где заказчик требует банковскую гарантию исполнения как обязательное условие. Без гарантии компания не сможет подписать договор и начать работы. Гарантия исполнения от Сбербанка оформляется за один день и закрывает требование заказчика, а кредитная линия на пополнение оборотных средств даст ресурс на закупку материалов и оплату подрядчиков.»

  Пример плохого обоснования (запрещено): «Логистической компании с регулярными контрактами нужны гарантии для обеспечения каждого нового договора и оборотный капитал для финансирования работ» — это констатация факта без объяснения ПОЧЕМУ. Не отвечает на вопрос: зачем гаранти? что будет без них? какую конкретную задачу решает продукт?

  Правильно: «Логистическая компания получила новый контракт на 12.5 млн ₽ — для госзаказчика банковская гарантия исполнения является обязательным требованием тендерной документации: без неё компания не может заключить договор и приступить к работам. Гарантия от Сбербанка с предварительным лимитом оформляется за один день, что позволяет обеспечивать каждый новый контракт без сбора документов каждый раз заново. Дополнительно компания несёт затраты на материалы и подрядчиков до получения оплаты по контракту — кредит на пополнение оборотных средств закрывает этот разрыв между расходами и поступлениями.»

  ⚠️ Проверка перед выводом: если из обоснования убрать название продукта — и при этом не будет понятно, ЗАЧЕМ компании этот продукт нужен в контексте события — обоснование НЕ годится. Переписать.

После таблицы — ключевые выводы по сегментам:
- **VIP (≥ 80%)**: список компаний — приоритет для персонального обслуживания. Действия: личная встреча с руководителем, предодобренное предложение, КП в течение 1 рабочего дня, подключение персонального менеджера.
- **Высокий (70–79%)**: список — отраслевые предложения. Действия: звонок руководителю или финдиректору, предложение отраслевого пакета (например, для строительства — гарантии + кредитование, для нефтегаза — кредитование + ВЭД), КП в течение 2–3 дней.
- **Средний (50–69%)**: список — требуют отраслевого подхода. Действия: уточняющий звонок для оценки реальной потребности, предложение базового продукта (гарантии или РКО), передача информации отраслевому менеджеру для дозревания клиента, повторный контакт через 1–2 недели.
- **Низкий (< 50%)**: список — гарантии или мониторинг. Действия: внесение в список наблюдения, предложение гарантий при появлении нового контракта, без активных продаж.
